# Home Screen widget for latest threads and unread important articles

## Objective

Ship a WidgetKit Home Screen widget that surfaces the user's top items at a
glance. The user adds the widget, picks one of two content sources — **Latest
threads** or **Unread important articles** — and the widget shows the top 5 of
that source, rotating one at a time every few minutes, each over the item's hero
image. Tapping the widget opens the app directly to the currently-shown item.

Two sizes: **small square** (`.systemSmall`) and **wide rectangle**
(`.systemMedium`). The large square (`.systemLarge`) and iPad `.systemExtraLarge`
are out of scope.

## Context

- App target: `AggregatorApp` (iOS 26, iPhone-only, `TARGETED_DEVICE_FAMILY=1`).
  The Xcode project is generated by xcodegen from `project.yml`.
- Reuses existing app code: `APIClient` (`AggregatorApp/Common/APIClient.swift`),
  models `Article`/`Thread`, and credential storage (`KeychainHelper`,
  `CredentialsStore`). These must become reachable from the widget extension.
- Auth: every API request needs the two Cloudflare Access headers, with values
  in the Keychain (`KeychainHelper` uses `kSecClassGenericPassword`). The widget
  runs in a **separate process** and currently cannot read them.

### WidgetKit constraints that shape this design (not optional)

- **No timers/animation.** A widget renders a *timeline* of dated entries; iOS
  swaps to the entry whose date is current. "Rotation" is therefore a timeline
  of 5 entries with staggered dates — the swaps cost no network/refresh budget.
- **Refresh budget.** iOS decides background refresh cadence (~15–60 min for a
  visible widget); it is not guaranteeable and not per-second. Data freshness is
  bounded by this; rotation within an already-built timeline is not.
- **Memory budget.** The widget render process is limited (~30 MB). Hero images
  must be downsampled before display; full-size images will crash the extension.
- **Background.** iOS 17+ widgets must declare their background via
  `.containerBackground(for: .widget)`.
- **Configuration** uses App Intents (`AppIntentConfiguration` +
  `WidgetConfigurationIntent`), per widget instance.

### Verified API behavior

- Threads: `GET /threads?sort=importance&limit=5` (cursor-paginated; first page
  is enough). `Thread` carries `representativeTitle`, `imageURL`, `lastUpdated`,
  `sourceCount`.
- Unread important: `GET /articles?view=important&unread_only=true&limit=5`
  (`docs/API.md:112`, `unread_only` per `docs/API.md:116`). `Article` carries
  `title`, `imageURL`, `sourceName`, `feedPublishedAt`, `id`, `isRead`.
- Both compose with `sort`. `APIClient` already builds these (`getThreads`,
  `getArticles(feed: .important, unreadOnly: true)`).

## Changes

### 1. App Group + shared Keychain (auth foundation) — build and verify FIRST

This is the prerequisite for everything else and the highest-risk piece
(entitlements + real signing). Build and verify it in isolation before the UI.

- Add an **App Group** (e.g. `group.net.renalias.AggregatorApp`) and a
  **Keychain Access Group** (e.g. `$(AppIdentifierPrefix)net.renalias.AggregatorApp.shared`)
  entitlement to **both** the app and widget targets, via `project.yml`
  (`entitlements` block per target; xcodegen writes the `.entitlements` files).
- Migrate `KeychainHelper` to read/write with `kSecAttrAccessGroup` set to the
  shared group, so the widget process can read the CF credentials. Preserve the
  existing service/account keys. Handle the one-time migration of any
  already-stored item into the shared group (read old → write to shared group).
- Verification bead: a tiny check (unit test or scratch code) proving the widget
  target can read the same credential the app wrote.

### 2. Shared code between app and widget

- Make `APIClient`, the networking/auth transport, `Article`, `Thread`,
  `PaginatedResponse`, `DateDisplay`, `CredentialsStore`/`KeychainHelper`
  reachable from the widget target via **shared source membership** in
  `project.yml` (the same files listed under both targets, e.g. a
  `AggregatorApp/Common` shared sources path), not a separate framework/SPM
  package. This is the simplest approach under xcodegen, keeps one source of
  truth, and must be decided **before** the entitlements work so the
  verification check (§1) links against the final shared structure.

### 3. Widget extension target

- New app-extension target `AggregatorWidget` in `project.yml`
  (`type: app-extension`, `platform: iOS`, deployment 26.0, same team,
  `PRODUCT_BUNDLE_IDENTIFIER: net.renalias.AggregatorApp.AggregatorWidget`),
  declared as a dependency of `AggregatorApp` so it embeds.
- `@main struct AggregatorWidgetBundle: WidgetBundle` exposing one widget kind
  `"AggregatorRadarWidget"`, supported families `[.systemSmall, .systemMedium]`,
  gallery display name + description.

### 4. Configuration intent

- `WidgetConfigurationIntent` with one `@Parameter` enum `ContentSource`:
  `latestThreads` (default) and `unreadImportant`, each `AppEnum` with a
  localized title. Per-instance; two widgets can show different sources.

### 5. Timeline provider (rotation + freshness)

- `AppIntentTimelineProvider` keyed on `ContentSource`:
  - Fetch the top 5 items for the selected source via `APIClient`.
  - For each item, download + **downsample** its hero image to the target
    family point-size (@3x), cache the result in the App Group container
    (keyed by item id + size); reuse cache across reloads. **On each timeline
    build, prune cached images not among the current 5 item ids** so the shared
    container does not grow unbounded.
  - Build **5 timeline entries** staggered ~3 min apart, entry *i* showing item
    *i* (with its prepared image + the deep-link URL).
  - `TimelineReloadPolicy`: `.after(now + ~30 min)` (after the rotation window),
    so data freshens ~every 30 min subject to system budget.
  - `placeholder(in:)` → redacted skeleton; `snapshot(...)` → bundled **sample
    data**, no network (for the gallery).

### 6. Widget views

- Small (`.systemSmall`): hero image as `containerBackground`, a dark bottom
  gradient **scrim** for legibility, title (≤2 lines) + source name.
- Medium (`.systemMedium`): hero + scrim, title (≤3 lines) + source · relative
  time (`DateDisplay`), importance tag if room.
- **No-hero fallback**: a bundled **radar-themed default image** (reusing the app
  icon's dark-navy radar motif) as the background. Add it as a widget asset.
- All text uses semantic colors/system fonts; ≥ minimum contrast over the scrim.

### 7. Deep linking + in-app routing

- Entries set `widgetURL(_:)` to `aggregator://article/{id}` or
  `aggregator://thread/{id}`. Tapping opens the *currently displayed* item.
- Register the `aggregator` URL scheme (Info.plist via `project.yml`
  `INFOPLIST_KEY_*` / `CFBundleURLTypes`).
- App handles `onOpenURL`: route to the relevant tab and push the item's detail
  (`ArticleDetailView` / `ThreadDetailView`). Must work from cold start
  (parse → set a navigation path the root consumes once views are ready).
- Opening an article marks it read via the existing flow; after the next widget
  reload an unread-important item that was read drops off naturally.

### 8. App-side widget reloads

- Call `WidgetCenter.shared.reloadTimelines(ofKind:)` after data/read changes
  that affect the widget (e.g. after marking read, after a foreground refresh),
  so the widget freshens sooner than the background budget alone allows.

### 9. States

State precedence (evaluated in this order):
- **Not configured** (CF credentials *absent* from the shared Keychain): widget
  shows an "Open app to sign in" prompt and makes **no** network attempt.
- **Empty** (credentials present, source returns 0 items): a friendly empty
  message.
- **Offline / fetch failure** (credentials present but the request fails — a
  network error *or* a 403 with an HTML body, which is a Cloudflare Access
  rejection, not an API error per CLAUDE.md): render the last good cached entries
  if present; otherwise a neutral "Can't update" state. Never blank.

Note a 403/HTML rejection means the stored credentials are invalid/expired, but
it is still treated as the error path (last-good / "Can't update"), not as
"Not configured" — only a genuinely absent credential drives the sign-in prompt.

### 10. Tests

- `APIClient` widget queries (threads top-5; `view=important&unread_only=true`
  top-5) produce the expected URLs (extend existing query tests).
- Image downsampling helper: given an oversized image + target size, returns an
  image within the target bounds.
- Deep-link parsing: `aggregator://article/123` / `aggregator://thread/45`
  → correct route value; malformed URLs → nil.
- Timeline builder: given 5 items, produces 5 entries with staggered, increasing
  dates and the right per-entry item + URL.
- ContentSource `AppEnum` default is `latestThreads`.
- Widget views are not unit-tested (use SwiftUI previews for visual checks).

## Acceptance Criteria

- [ ] `xcodebuild test` exits 0 incl. new widget-logic tests; app + widget build
- [ ] Widget appears in the gallery with a working preview (snapshot sample data, no network) in small and medium
- [ ] Long-press → Edit Widget lets the user choose **Latest threads** or **Unread important articles**; each instance honors its own choice
- [ ] Widget shows the top 5 of the selected source, rotating one at a time a few minutes apart (timeline entries), and the data reloads on the order of every ~30 min (system-budget permitting) and on app-triggered reloads
- [ ] Each item renders over its hero image with a legible scrim; items with no hero show the bundled radar default image
- [ ] Tapping the widget opens the app directly to the currently-shown article/thread, including from cold start
- [ ] The widget reads CF credentials from the shared Keychain Access Group; with no credentials it shows "Open app to sign in" and makes no network call
- [ ] Empty and offline/stale states are handled (last-good data shown when available; never blank)
- [ ] Hero images are downsampled; the widget does not exceed its memory budget (no render crashes) for large images
- [ ] The App Group image cache is pruned to the current 5 item ids on each build (does not grow unbounded across reloads)
- [ ] No hardcoded hex colors; semantic colors/system fonts; tap target is the whole widget

## Pending Decisions

- **Content sources**: resolved — *Latest threads* (`GET /threads`, top 5) and
  *Unread important articles* (`GET /articles?view=important&unread_only=true`,
  top 5). Default source `latestThreads`.
- **No-hero fallback**: resolved — bundled **radar-themed default image** (app
  icon motif), not a flat color.
- **Surfaces**: resolved — Home Screen **small + medium** only. Lock
  Screen/StandBy accessory widgets and iPad `.systemExtraLarge` are out of scope
  for v1 (possible follow-up).
- **Rotation / refresh**: resolved — 5 entries ~3 min apart in one timeline;
  reload ~every 30 min + app-triggered `WidgetCenter` reloads. Per-second
  freshness is not possible on iOS and is not a goal.
- **Out of scope (v1)**: interactive buttons, multiple items visible at once,
  showing more than 5 items, per-source color theming.
- **Open — App Group / Keychain group identifiers**: exact strings
  (`group.net.renalias.AggregatorApp`, the keychain group) to be finalized at
  implementation; must match across both targets' entitlements.
- **Shared-code mechanism**: resolved — **shared source membership** in
  `project.yml` (same files under both targets), not a framework/SPM package.
  Decided up front because the §1 entitlements/verification work depends on it.
